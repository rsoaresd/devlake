# AgentReady Plugin

## Overview

The AgentReady plugin collects and analyzes AI readiness assessments from repositories. It fetches structured assessment JSON files from GitHub or GitLab, extracts findings with tiered pass/fail results, and calculates aggregate metrics. Three Grafana dashboards provide fleet-level, findings-level, and per-repo visibility.

Assessments are generated by [AgentReady](https://github.com/ambient-code/agentready), an open-source tool that evaluates git repositories for AI-assisted development readiness. AgentReady scores repositories across 13 evidence-based categories and produces certification levels (Platinum, Gold, Silver, Bronze). This plugin ingests those assessment outputs into DevLake for centralized tracking and trend analysis across your fleet.

### Repository Setup For Tracking

To track a repository in this plugin it **must have the following JSON files**:
- `.agentready/assessment-latest.json`
- `.agentready/assessment-<YYYYMMDD>-<HHMMSS>.json`

All other files can be ignored in a `.gitignore`:
```
.agentready/*.html
.agentready/*.md
```

Repositories without these files **are ignored**.

The `.agentready/assessment-latest.json` file should point to the latest assessment. It gets updated every time an agentready assessment is run.

## Plugin Architecture

This is a **metric/transformer plugin** (not a data source plugin):

- Works on domain layer data from GitHub/GitLab plugins
- Implements `MetricPluginBlueprintV200` interface
- Configured via Projects, not Connections
- Runs after data collection plugins

## Features

- **Multi-platform support**: Works with both GitHub and GitLab repositories
- **Tiered assessment model**: Findings categorized into 4 tiers (Essential, Critical, Important, Advanced)
- **Certification levels**: Overall scores mapped to certification levels (e.g., Gold, Silver)
- **Fleet-level analytics**: Aggregate statistics across all repositories in a project
- **Trend tracking**: Pass rates per tier over time for compliance monitoring
- **REST API**: Query assessments, findings, and statistics programmatically
- **Grafana dashboards**: Three pre-built dashboards for fleet, findings, and repo-level views

## Data Models

### Assessment

Stores assessment records fetched from repositories:
- Repository identity (ID, name, provider, connection)
- Schema version and overall score
- Certification level
- Attributes assessed vs. total available
- Branch, commit hash, and assessment timestamp
- Duration and raw JSON payload

Primary key format: `{repoId}:{commitHash}`

### Finding

Individual findings extracted from each assessment:
- Attribute identity (ID, name, category)
- Tier (1=Essential, 2=Critical, 3=Important, 4=Advanced)
- Status (pass, fail, skipped, error)
- Score, measured value, and threshold
- Evidence, remediation summary, and remediation steps
- Default weight for scoring

Primary key format: `{assessmentId}:{attributeId}`

### Metric

Aggregated metrics computed per assessment:
- Pass, fail, and skip counts
- Per-tier pass rates (Tier 1 through Tier 4) as percentages
- Category scores (JSON map of category to average score)

Primary key format: `{repoId}:{timestamp}:{assessmentId}`

## Subtasks

The plugin pipeline has three stages:

1. **collectAssessments**: Discovers repos from a project or single repoId, fetches assessment JSON files from GitHub/GitLab APIs, and stores raw JSON. Repos without an assessment file are skipped (not treated as errors).

2. **extractAssessments**: Parses raw JSON into structured assessment and finding records. Findings with status `not_applicable` are excluded. Depends on `collectAssessments`.

3. **calculateMetrics**: Aggregates findings per assessment into pass/fail/skip counts, tier pass rates, and category scores. Depends on `extractAssessments`.

## API Endpoints

All endpoints support filtering via `projectName` and/or `repoId` query parameters.

### Assessments

| Method | Path | Description |
|--------|------|-------------|
| GET | `/assessments` | List assessments with pagination. Filters: `projectName`, `repoId`, `certification`, `page`, `pageSize` |
| GET | `/assessments/:id` | Get a single assessment with its findings |
| GET | `/assessments/:id/findings` | Get findings for an assessment. Filters: `tier`, `status`, `category` |

### Statistics

| Method | Path | Description |
|--------|------|-------------|
| GET | `/stats` | Aggregated stats: `totalRepos`, `averageScore`, `certificationDistribution` |

### Scope Configs

| Method | Path | Description |
|--------|------|-------------|
| GET | `/scope-configs` | List all scope configs |
| POST | `/scope-configs` | Create a scope config |
| GET | `/scope-configs/:id` | Get a scope config |
| PATCH | `/scope-configs/:id` | Update a scope config |
| DELETE | `/scope-configs/:id` | Delete a scope config |

## Configuration

### Scope Config Options

```json
{
  "branch": "main",
  "assessmentFilePath": ".agentready/assessment-latest.json",
  "excludeRepos": "repo1,repo2"
}
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `branch` | string | (default branch) | Git branch to fetch assessment from. Empty uses the repository default branch |
| `assessmentFilePath` | string | `.agentready/assessment-latest.json` | Path within the repository to the assessment JSON file |
| `excludeRepos` | string | (none) | Comma-separated list of repository names to skip |

## Usage

### Prerequisites

This plugin requires data from either the GitHub or GitLab plugin. Ensure you have:
1. Collected repository data using the respective plugin
2. Assessment JSON files committed to your repositories

### Assessment JSON Format

The plugin expects a JSON file in each repository at the configured path (default: `.agentready/assessment-latest.json`):

```json
{
  "schema_version": "1.0",
  "repository": {
    "name": "my-repo",
    "branch": "main",
    "commit_hash": "abc123"
  },
  "timestamp": "2024-01-15T10:30:00Z",
  "overall_score": 85.5,
  "certification_level": "Gold",
  "attributes_assessed": 15,
  "attributes_total": 20,
  "duration_seconds": 123.45,
  "findings": [
    {
      "attribute": {
        "id": "sec-scanning",
        "name": "Security Scanning",
        "category": "Security",
        "tier": 1,
        "default_weight": 0.1
      },
      "status": "pass",
      "score": 95.0,
      "measured_value": "enabled",
      "threshold": "enabled",
      "evidence": ["SAST configured in CI pipeline"],
      "remediation": {
        "summary": "Enable security scanning",
        "steps": ["Add SAST stage to CI config"]
      }
    }
  ]
}
```

### Running the Plugin

Configure it in your DevLake blueprint:

```json
{
  "plugin": "agentready",
  "options": {
    "projectName": "my-project",
    "scopeConfig": {
      "branch": "main",
      "assessmentFilePath": ".agentready/assessment-latest.json"
    }
  }
}
```

Or target a single repository:

```json
{
  "plugin": "agentready",
  "options": {
    "repoId": "github:GithubRepo:1:12345"
  }
}
```

## Database Tables

- `_tool_agentready_assessments`: Assessment records with scores and metadata
- `_tool_agentready_findings`: Individual findings per assessment
- `_tool_agentready_metrics`: Aggregated pass rates and category scores
- `_tool_agentready_scope_configs`: Per-scope configuration

## Grafana Dashboards

### Fleet Overview (`fleet-overview.json`)
Aggregate view across all repositories: average scores, certification distribution, and fleet-level trends.

### Findings Analysis (`findings-analysis.json`)
Top 15 most common failures (bar chart) and tier pass rates over time (time series). Useful for identifying systemic gaps.

### Repository Detail (`repo-detail.json`)
Single-repo deep dive with current score, certification level, and detailed findings breakdown. Filtered by `${repo_id}` variable.

## Tier Model

| Tier | Name | Description |
|------|------|-------------|
| 1 | Essential | Baseline requirements every repository must meet |
| 2 | Critical | Important practices for production readiness |
| 3 | Important | Recommended practices for mature repositories |
| 4 | Advanced | Best-in-class practices for leading repositories |
